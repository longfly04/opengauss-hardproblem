# TPC系列Benchmark

事务处理性能委员会（Transaction Processing Performance Council, TPC）自1988年成立以来，制定了一系列基准测试标准，为数据库系统性能评估提供了权威框架。这些标准包括面向联机事务处理（OLTP）的TPC-C、TPC-E，面向决策支持系统（DSS）的TPC-H、TPC-DS，以及针对新兴技术的TPCx-IoT、TPCx-AI等。

## 一、TPC系列标准核心特征

**TPC-C（1992年发布）**是数据库领域最持久、最广泛使用的基准测试之一，专门针对OLTP系统设计。其核心特征包括：

- **模拟场景**：简单批发商系统，包含仓库、客户、订单等基本业务实体
- **数据库结构**：9张表，包含客户、订单、库存等关系
- **事务类型**：5种事务（新订单、支付、订单状态查询、发货、库存水平检查）
- **性能指标**：tpmC（每分钟处理的新订单事务数），以及性价比指标$/tpmC
- **设计原则**：遵循Jim Gray提出的基准测试四原则（相关性、可移植性、可扩展性、简单性）

TPC-C通过严格的标准化和可重复性要求，成为衡量OLTP系统性能的黄金标准。其设计考虑了多种事务并发执行、在线与离线交易执行模式、大量磁盘I/O数据流等实际应用场景。然而，随着技术发展，TPC-C的局限性逐渐显现，尤其是其简单性无法反映现代高并发场景的复杂性。

**TPC-H（1999年发布）**是面向决策支持系统（DSS）和数据仓库的权威基准测试，其核心特征包括：

- **模拟场景**：商业分析行为，包含22条复杂SQL查询
- **数据库结构**：8个基本关系，采用第三范式（3NF）建模
- **数据规模**：从1GB到100TB不等，可扩展性强
- **性能指标**：QphH（每小时查询数），版本2表示为Price/QphH@Size，版本3表示为Price/kQphH@Size
- **测试特点**：强调执行查询时的数据库大小、单个数据流提交查询能力以及并发用户查询吞吐量

TPC-H通过复杂关联分析（如多表JOIN、相关查询、数据多维过滤等），为OLAP系统性能评估提供了全面框架。然而，随着数据量的指数级增长和分析需求的多样化，TPC-H在数据规模、实时分析支持和非关系型数据库适配方面逐渐显现出不足。


**TPC-E（2007年发布）**作为TPC-C的升级版，针对现代OLTP系统设计，其核心特征包括：

- **模拟场景**：复杂证券交易所系统，模拟金融信息服务环境
- **数据库结构**：33张表，包含客户账户、持仓、交易记录等
- **事务类型**：12种不同事务类型，涉及更多表连接、复杂查询和引用完整性约束
- **性能指标**：tpsE（每秒事务数），以及性价比指标Price/tpsE
- **数据生成**：基于真实数据规律，有数据倾斜（skew），而非随机均匀分布

TPC-E通过提升事务复杂度（如读写比例从TPC-C的1.9:1提高到9.7:1）、增加表数量（从9张到33张）和引入真实数据分布，试图更真实地模拟现代OLTP系统。然而，TPC-E发布后并未完全取代TPC-C，厂商接受度存在显著差异，微软成为早期唯一积极提交TPC-E测试结果的主要数据库厂商。

**TPC-DI（2006年发布）**是针对数据集成和ETL过程的基准测试，其核心特征包括：

- **模拟场景**：数据集成与转换负载，模拟数据仓库的ETL过程
- **测试重点**：数据转换、清洗、加载和集成效率
- **性能指标**：记录转换率（如每秒转换的记录数）

TPC-DI填补了数据集成领域的评测空白，但其应用范围相对有限，主要面向ETL工具而非通用数据库系统。

**TPCx系列**是TPC在2010年后推出的一系列新兴技术领域的基准测试，包括：

- **TPCx-HS**：面向混合分析与事务处理（HTAP）的工作负载
- **TPCx-V**：面向虚拟化环境的性能测试
- **TPCx-BB**：面向区块链应用的基准测试
- **TPCx-HCI**：面向超融合基础设施的性能评估
- **TPCx-IoT**：面向物联网数据处理的基准测试
- **TPCx-AI**：面向AI负载集成的数据库性能测试（尚在发展中）


## 二、TPC标准的关系模型

**TPC-C的事务模型**设计简单但具有代表性，包含五种事务类型，其比例固定为：新订单（≤45%）、支付（≥43%）、订单状态查询（≥4%）、发货（≥4%）和库存水平检查（≥4%）。数据生成采用随机数均匀分布方式，虽然实现简单，但缺乏真实业务场景中的数据倾斜。

**TPC-E的事务模型**显著复杂化，包含十二种不同类型和复杂度的并发事务，这些事务可以在线执行，也可以由价格或时间条件触发。数据库由三十三个表组成，包含各种列、基数和扩展属性。数据生成遵循真实业务规律，引入了数据倾斜，更接近实际应用场景。

**TPCx-IoT的数据生成**特别针对物联网场景设计，模拟大规模设备数据的采集、存储、查询与分析。其核心挑战在于高并发写入与实时查询的平衡，以及海量时序数据的压缩存储效率。

## 三、现有TPC系列基准的局限性

**TPC-C的局限性**：

1. **场景简单性**：批发商系统模型已无法反映现代OLTP应用的复杂性，如金融、电商等领域的高频交易和复杂业务逻辑
2. **事务模型陈旧**：5种事务类型和9张表结构设计过于简单，缺乏真实业务中的数据倾斜和复杂JOIN操作
3. **并发处理能力不足**：TPC-C的事务混合比固定，无法模拟真实业务中的突发流量和资源竞争


**TPC-E的局限性**：

1. **实现成本高**：TPC-E的33张表和12种事务类型设计导致测试系统构建成本显著增加，厂商需投入更多资源进行优化
2. **厂商策略差异**：IBM、戴尔等厂商虽提交了TPC-E测试结果，但Oracle、SQL Server等主流数据库仍主要采用TPC-C进行性能展示
3. **测试结果实用性**：TPC-E的测试结果是否能够真实反映实际应用场景中的性能表现，学术界存在分歧


**TPC-H的分析能力局限性**：

1. **数据规模限制**：虽然支持到100TB数据，但对于处理PB级数据的现代数据仓库仍显不足
2. **查询负载单一**：22条查询无法覆盖所有可能的分析场景，尤其缺乏实时分析和复杂图查询支持
3. **非关系型数据库适配不足**：TPC-H主要针对关系型数据库设计，对NoSQL、向量数据库等新兴架构支持有限



**TPC标准在云环境中的局限性**：

1. **动态资源分配**：传统TPC标准基于静态负载设计，无法评估云环境中自动扩缩容的能力
2. **多云架构兼容性**：跨云平台的性能一致性评估在现有标准中缺乏支持
3. **定价模型差异**：TPC传统的固定价格模型与云服务按需付费模式存在根本性冲突


## 四、现阶段研究空白

尽管已有显著进展，根据文献分析，仍存在以下研究机会：

### 1. 微观基准的标准化缺失：

**现状：**：目前极度缺乏开源的HTAP微观基准。现有的宏观基准难以精确控制读写干扰和数据生成速率。

**机会：**开发标准化的微观基准工具集，用于精准测量特定组件（如索引机制、MVCC版本链遍历）在HTAP环境下的性能表现。

### 2. 数据新鲜度的量化难题： 

**现状：**虽然HyBench和HATtrick都引入了Freshness指标，但测量方法（如基于时间戳差值）在不同一致性级别（Snapshot Isolation vs. Serializable）下存在偏差。

**机会：**研究非侵入式的、细粒度的Freshness测量技术，特别是在涉及复杂ETL逻辑或流处理集成的现代HTAP系统中。

### 3. 动态负载与异常处理的评估：

**现状：**大多数基准测试假设负载是平稳的。HyBench引入了“异常生成”（如欺诈检测），但这方面的研究还很少。

**机会：**未来的基准测试应包含动态数据分布（Dynamic Data Distribution）和实时风险控制场景的评估，以模拟真实的数字化转型需求。

### 4. 云原生与存算分离架构的评估：

**现状：**随着Snowflake、AlloyDB等云原生数据库的兴起，传统的基于本地存储的基准测试（如I/O模式）不再适用。

**机会：**设计专门针对“存储与计算分离”架构的HTAP基准测试，重点评估跨网络的数据传输延迟对实时分析的影响。


## 五、构造难题场景

### 难题1. 内存池自适应管理技术测试场景

**1. 大行对账场景测试（基于TPC-H）**
- **数据准备**：使用TPC-H生成10GB规模的数据集，模拟银行对账场景的数据量
- **查询构造**：选择TPC-H中包含大内存使用算子的查询（如Q1、Q3、Q6、Q10等），这些查询涉及大量排序、哈希连接操作
- **测试流程**：
  - 阶段1：使用默认内存参数配置，执行选定的TPC-H查询，记录落盘次数和执行时间
  - 阶段2：启用内存池自适应管理，执行相同查询，记录落盘次数和执行时间
  - 阶段3：对比两个阶段的结果，验证落盘降低90%以上，性能提升10X的目标

**2. 高并发TP流量与慢SQL混合场景测试（基于TPC-C和TPC-H）**
- **环境配置**：16U/32GB低规格内存配置
- **数据准备**：
  - 使用TPC-C生成数据集，模拟高并发交易场景
  - 使用TPC-H生成数据集，提供慢SQL查询源
- **测试流程**：
  - 阶段1：使用sysbench基于TPC-C模型构建高并发TP流量（500-1000并发）
  - 阶段2：在TP流量稳定后，瞬间注入5-10个包含大内存使用算子的TPC-H查询
  - 阶段3：监控TPS变化，验证内存池自适应调整后，TPS抖动控制在10%以内

### 难题2：内存动态调整和优雅回收技术测试场景

**1. 高并发TP流量与慢SQL混合场景测试（基于TPC-C和TPC-H）**
- **环境配置**：8U64GB规格以上服务器
- **数据准备**：
  - 使用TPC-C生成数据集，模拟高并发交易场景
  - 使用TPC-H生成数据集，提供慢SQL查询源
- **测试流程**：
  - 阶段1：使用sysbench基于TPC-C模型构建高并发TP流量（300-500并发）
  - 阶段2：在TP流量稳定后，快速注入3-5个包含大内存使用算子的TPC-H查询
  - 阶段3：监控TPS变化，验证内存动态调整后，TPS抖动控制在10%以内

**2. 慢SQL占用内存超90%场景下的并发能力测试（基于TPC-C和TPC-H）**
- **环境配置**：8U64GB规格以上服务器
- **数据准备**：
  - 使用TPC-C生成数据集，模拟高并发交易场景
  - 使用TPC-H生成数据集，提供慢SQL查询源
- **测试流程**：
  - 阶段1：执行多个TPC-H复杂查询，使内存占用达到90%以上
  - 阶段2：在内存高占用状态下，注入高并发sysbench TPC-C业务（300-500并发）
  - 阶段3：测量sysbench TPS，验证是否达到无背景慢SQL场景的90%以上

### 3. 测试指标与监控

**核心指标**：
- TPS（每秒事务数）：衡量OLTP性能
- 落盘次数：衡量内存使用效率
- 执行时间：衡量查询性能
- TPS抖动：衡量内存池调整对系统稳定性的影响
- 内存使用分布：监控各内存池的使用情况

**监控工具**：
- 数据库内置性能视图：查看内存使用、算子执行情况
- 系统监控工具：监控CPU、内存、I/O等系统资源使用
- 自定义监控脚本：记录TPS变化、查询执行时间等指标

通过以上基于TPC系列基准的测试场景，我们可以全面验证opengauss数据库的内存池自适应管理技术和内存动态调整与优雅回收技术的效果，确保其满足技术诉求中提出的各项指标要求。

### 4. 注意事项

**TPC系列benchmark**仅模拟在TP类负载下，注入大查询的AP负载时内存池的调度行为，与真实的环境依旧存在差别：
- TPCC 和 TPCH 负载之间使用的是不同的关系模式，所以TPCC的事务与TPCH的事务是不同的，无法模拟真实业务中“数据更新之后立即分析”这种场景
- 但是可以用于模拟难题2中“高并发的TP流量，同时瞬间注入包含大内存使用算子，启动内存池自适应调整”和难题3中“工作内存bound自适应调整和优雅回收”的场景
- 为了更加贴近真实，建议增加使用HTAP原生的混合事务Benchmark，比如HyBench、HTAPBench等进行测试的用例
